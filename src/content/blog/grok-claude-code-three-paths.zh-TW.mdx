> 這是上一篇的續集。上回寫到「想在 Claude Code 裡把 Grok 的訂閱吃進來，卻卡死在最後一步 OAuth」就停住了。這篇把懸念拆開——路有三條、誤會有一個，而真正要帶走的，是「任務怎麼分」這件事。**本文初稿由 Grok 4.5 自己撰寫**（我請它寫一篇「怎麼在 Claude Code 裡用 Grok」的文，作者就是它本人），我做潤色編輯。

上一篇寫到最後，故事停在一個很尷尬的地方：想在 Claude Code 裡把 Grok 的訂閱吃進來，結果卡死在 OAuth。畫面跳出來要授權，工具卻怎麼也接不上，像是隔著一層玻璃看著能用卻用不了的東西。當時我以為自己已經走到死胡同，關掉編輯器去睡覺，心裡還在盤算要不要乾脆放棄。

後來才發現，那層玻璃有一半是我自己糊上去的。

## 一、診斷錯誤，繞了一大圈

最初我認定卡關的原因是「團隊帳號」。官方文件與社群討論裡，團隊方案對第三方登入常有額外限制，我很自然地把責任推給帳號類型，開始搜各種繞過方式、看誰有類似經驗。結果花了不少時間，才在帳號後台確認：這其實是個人訂閱。

一個簡單的事實誤判，讓後面的路徑選擇全部偏掉。這不是什麼高深教訓，卻很實用——當工具登不進去時，先把「我以為的前提」一條條核對，往往比立刻鑽技術細節更有效。誤把個人訂閱當團隊方案，等於一開始就站在錯誤的地圖上找路。

釐清這點之後，局面反而打開了。我發現並不是「只能卡在 OAuth 死掉」，而是有三條明顯不同的路，可以在 Claude Code 裡、或緊鄰它的位置，把 Grok 用起來。每條路的取捨完全不一樣。

## 二、三條路，各有代價

**第一條路**，是透過開源中轉工具，把訂閱授權「借」過來用。概念上就是讓訂閱的憑證在背景被某個中介層拿去呼叫模型，於是你在 Claude Code 的對話裡就能直接叫到 Grok，費用走訂閱而不是按量。好處顯而易見：有訂閱的人省錢。壞處也清楚——這落在官方明確認可的第一方路徑之外，屬於灰色地帶。官方只保證自己的登入與介面，這類中轉隨時可能因為政策收緊而失效，憑證也可能過期或被撤銷。我刻意不寫任何實作細節，因為它本來就不該被當作穩定方案來推廣。

**第二條路**，是用官方 API 金鑰，再搭配中轉層接到 Claude Code。這條路走的是按量計費，沒有「吃訂閱」的快感，但實際算下來其實很便宜，而且完全在官方允許的範圍內。沒有灰色、不需要背景續命程式、也不怕帳號類型扯後腿。只要金鑰有效，誰都能用，穩定性最高。

**第三條路**，是直接用官方提供的命令列工具登入。這是第一方最乾淨的做法：OAuth 在它自己的流程裡完成，介面也是官方維護的。缺點只有一點——它不是留在 Claude Code 裡，而是換成另一套命令列體驗。如果你執著「一定要在同一個編輯器視窗裡打字」，這條路就不符合；如果你只在乎模型本身能幫忙寫 code、除錯、想架構，那它反而最省心。

## 三、怎麼選，取決於你要什麼

想穩、想簡單、不想半夜被通知「授權失效」——走 API 金鑰那條。已有訂閱、預算敏感、又願意接受灰色風險——可以試訂閱中轉，但心裡要清楚它不是地基。只想讓模型幫忙寫 code、不執著介面必須是 Claude Code——官方命令列工具最乾淨。

這三條路沒有絕對的「最強」，只有「適不適合當下的任務」。

## 四、真正的心得：不是工具，是任務分配

寫到這裡，最大的收穫反而不是「終於找到方法了」，而是重新想清楚一件事：怎麼分配任務。

一句話就能說完——**無人值守的自動化，走穩定的計費路（API 金鑰）；人在場的互動，才走省錢的訂閱路。**

為什麼？因為訂閱那條路有三層脆弱。第一，憑證會過期，而且過期時間不一定跟你預期的一樣。第二，它通常需要背景程式或中介服務持續「續命」，這些東西本身就可能掛掉、被系統殺掉、或在重開機後沒自動起來。第三，灰色名單隨時可被收緊，今天能用的中轉，明天可能直接被擋。把這種東西拿來當半夜沒人盯著的自動化地基，等於把整個流程建立在會自己崩塌的沙地上。人在場時，授權跳出來你可以立刻處理；人睡著時，失敗就是失敗，沒有補救窗口。

工具不是拿來炫技的。能同時接三個模型、能吃訂閱、能在同一個視窗裡切換——這些都很酷，但酷不代表可靠。真正有用的，是讓每個工具做它擅長且風險可控的事：穩定的留給自動化，省錢的留給人機協作。分配清楚，半夜才睡得著。

## 五、最後講清楚一個邊界

「能用」不等於「免費」。訂閱中轉那條路，我到現在都還沒完整對過帳，不敢保證背後完全沒有偷偷計費或額度消耗。有些服務表面上走訂閱，實際呼叫時仍可能觸發額外計量。在沒有自己驗證之前，我不會跟任何人說「這招穩賺」。省錢是目標，但前提是帳算得清楚；算不清楚的省，最後可能變成另一種成本。

上一篇卡在 OAuth 的懸念，到這裡算是拆開了。路有三條，誤會有一個，真正要帶走的，是「任務怎麼分」這件事。工具會換、政策會改，但把自動化放在穩的路上、把互動放在省的路上，這條判斷大概還能用一陣子。
